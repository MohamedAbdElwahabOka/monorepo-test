الأداة تكتب النتيجة في الملف

الملف generated/dart/booking_model.dart يتم الكتابة فوقه تلقائياً بالحقل الجديد.

الخلاصة للمطور الجديد: أنت مش هتكتب أي سكريبتات معقدة! أنت بس هتروح لملف openapi.yaml، تضيف الحقل الجديد، وتشغل أمر npm run generate. الأداة (OpenAPI Generator) هتاخد الـ YAML وتصب البيانات في قوالب جاهزة، وتطلعلك الكود النظيف الجاهز في كل المشاريع. أنت مطور، مش صانع أدوات!

كيف تفهم الأداة ملف الـ YAML وتتكيف مع كل مشروع؟

الأداة (مكتوبة بـ Java) ليست ذكاءً اصطناعياً. هي تقوم بتحليل الـ YAML (Parsing) وتحوله لشجرة بيانات (AST). بعد ذلك تستخدم ما يسمى بـ "Generators" محددة (مثل typescript-axios للـ Backend أو dart للموبايل). كل Generator يمتلك مجموعة قوالب (Mustache Templates) يأخذ البيانات ويصبها فيها ليخرج الكود المناسب للغة المشروع.

هل أقوم بالتعديل دائماً في الـ YAML؟ وكيف أضيف شرطاً (مثل: السن >= 12)؟

نعم، التعديل دائماً يبدأ من ملف openapi.yaml لأنه الـ Single Source of Truth.

لإضافة شرط السن، تضيف minimum: 12 تحت الحقل في الـ YAML. وعندما تقوم بتشغيل الـ Generator، تقوم الأداة بترجمة هذا الشرط إلى Annotations في الـ Backend (مثل @Min(12) في NestJS) وإلى Validation rules في كود الموبايل، وذلك بناءً على ما تمت برمجته مسبقاً داخل القوالب (Templates).

أين يذهب الكود المولّد؟ ومن يكتب هذه القوالب (Templates)؟

الكود المولّد لا يكتب الـ Business Logic الخاص بك (مثل Controllers أو Services)، بل يقوم بتوليد هياكل البيانات (Interfaces و DTOs) فقط. أنت تأخذ هذه الملفات المولّدة وتقوم بعمل Import لها لتستخدمها في الـ Controller الخاص بك.

من يكتب القوالب؟ الأداة تأتي بآلاف القوالب الافتراضية الجاهزة (مفتوحة المصدر)، ولكن إذا احتاج فريقك (مثل الـ DevOps أو الـ Senior) شيئاً مخصصاً لـ Rihla، يمكنهم كتابة Custom Template خاص وإجبار الأداة على استخدامه بدلاً من الافتراضي.

هل معنى ذلك أن هذا النظام كله معمول لتغييرات الداتا (Data) وليس لـ (Business Logic) المشروع؟

بالضبط! أنت لخصت الفكرة 100%.

هذا النظام (Code Generation) مسؤول فقط عن "عقد الاتصال" (The Contract). هو يولد شكل البيانات (Models, DTOs)، قواعد التحقق (Validation)، ودوال الاتصال بالشبكة (Network Clients) مثل createBooking().

ماذا عن اللوجيك؟ الأداة لا تعرف كيف تحسب خصم الـ Subscription، ولا كيف تتواصل مع بوابات الدفع (PayMob)، ولا كيف ترسل إشعار WhatsApp. هذا اللوجيك (Business Logic) تكتبه أنت بنفسك داخل الـ Services الخاصة بك.

الهدف الحقيقي: الأداة تخلصك من كود "النسخ واللصق" الممل بين المشاريع وتضمن عدم وجود أخطاء في نقل البيانات، عشان توفر وقتك وتخليك تركز على كتابة "اللوجيك" المعقد والمهم!

ماذا تقصد بـ "دوال الاتصال بالشبكة (Network Clients)" مثل createBooking()؟

في الطريقة القديمة، عشان سارة (مطورة الموبايل) تبعت حجز للباك إند، كانت بتضطر تكتب كود طويل وممل: تجهز رابط الـ API، وتعمل HTTP POST request، وتظبط الـ Headers (زي الـ Token)، وتعمل تحويل للبيانات من JSON لـ Dart Object والعكس (Serialization).

مع الـ Code Generation: الأداة بتقرأ من الـ YAML إن فيه مسار اسمه /bookings نوعه POST، فتقوم مولدة كلاس كامل اسمه API Client جواه دالة جاهزة اسمها createBooking().

سارة كل اللي بتعمله إنها بتنادي الدالة دي في سطر واحد:
var response = await apiClient.createBooking(bookingData);
والأداة بتكون كاتبة جوه الدالة دي كل كود الـ HTTP، والـ Headers، والـ JSON Parsing! ده بيوفر ساعات من كتابة الكود الروتيني (Boilerplate) وبيمنع أخطاء الـ (Typos) في أسماء الروابط أو الحقول.

📚 مسرد المصطلحات (للمبتدئين)

📄 Single Source of Truth

المصدر الوحيد الموثوق للمعلومات. في Rihla، هو ملف openapi.yaml اللي بيوصف كل الموديلات والـ APIs.

🤖 Code Generation

عملية توليد الكود تلقائياً من ملف تعريف. زي ما تكون عندك ماكينة مطبعة: تديها القالب والبيانات، تطلعلك كتاب جاهز!

🌳 AST (Abstract Syntax Tree)

طريقة الكمبيوتر في فهم الكود. الأداة بتحول الـ YAML لشكل شجري (زي شجرة الملفات اللي فوق) عشان تفهمه وتتعامل معاه.

📦 Mustache Templates

قوالب فيها أماكن فاضية {{ }}، الأداة بتملأها بالبيانات. زي ما يكون عندك نموذج (Form) في الورق وبتملأ الخانات.

⚡ Turborepo Cache

تخزين مؤقت لنتائج البناء. لو سارة عملت Build قبل كده ومفيش تغييرات جديدة، مش هتضيع وقت تعيده تاني — بياخد ثانية!

🔄 CI/CD Pipeline

سلسلة خطوات أوتوماتيكية بتشتغل على السيرفر (زي GitHub Actions) كل ما حد يرفع كود. بتتأكد إن الكود مفيهوش أخطاء قبل ما يتدمج.

✅ خطواتك كـ Junior عند إضافة حقل جديد

1 افتح ملف packages/api-contract/openapi.yaml.
2 أضف الحقل الجديد تحت properties مع تحديد النوع (string, number, integer).
3 احفظ الملف وارفع التغييرات (أو شغّل محلياً).
4 شغّل الأمر npm run generate.
5 تأكد من ظهور الكود الجديد في مجلدات generated/.
6 اعدل الكود في المشاريع (Backend Service / Flutter UI) عشان تستخدم الحقل الجديد.
7 ارفع Pull Request وانتظر نجاح CI/CD (الـ Pipeline الأخضر ✅).

❓ أسئلة المطورين الجدد

ماذا لو غلطت في كتابة الـ YAML؟

الأداة هتطلعلك رسالة خطأ واضحة (زي Invalid YAML at line 45)، والـ CI/CD هيرفض الدمج. هتروح تعدل الغلطة وتجرب تاني.

هل لازم أتعلم Mustache Templates؟

لا طبعاً! القوالب دي مسؤولية فريق الأدوات (DevOps). أنت كمطور عادي بتستخدم القوالب الجاهزة وخلاص.

إزاي أتأكد إن التعديل وصل لكل المشاريع؟

شوف مجلد generated/ في كل مشروع بعد التوليد. لو الملفات اتحدثت (git diff يوضحلك)، يبقى وصل.

ليه الـ Build بياخد وقت طويل أول مرة؟

في أول مرة بس (Cold Build) عشان بيبني كل حاجة من الصفر. بعد كده Turborepo بيستخدم الـ Cache ويخلي البناء في ثواني.

إيه الـ npm run generate بيعمل بالضبط؟

ده أمر مختصر (alias) في ملف package.json بيشغل أداة openapi-generator-cli generate مع إعدادات المشروع.

لو عاوز أضيف حقل required مش optional؟

في ملف الـ YAML، ضيف اسم الحقل في قائمة required اللي في أول الموديل. كده الكود المولّد مش هيحط ? (nullable).
